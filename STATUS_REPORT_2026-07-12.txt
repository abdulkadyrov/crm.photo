Отчет по текущему состоянию проекта Vakha Studio
Дата: 2026-07-12

1. Что это за проект сейчас

Проект представляет собой офлайн-ориентированное PWA-приложение для управления школьными фото- и видеосъемками. Приложение запускается как статический фронтенд без серверной части и хранит рабочие данные локально в IndexedDB браузера.

По структуре репозитория видно, что проект уже прошел несколько этапов развития:
- есть PWA-оболочка с manifest и service worker;
- есть вынесенные папки js/core, js/modules, js/services, js/ui, js/utils;
- при этом реальная бизнес-логика до сих пор в основном сосредоточена в одном большом файле js/core/legacy-app.js;
- корневой app.js сейчас используется как совместимость со старым кешем, а основная точка входа идет через js/core/app.js.

Итогово: проект не выглядит пустым или сырым прототипом. Функционально он уже насыщенный. Но архитектурно он все еще находится в промежуточном состоянии между монолитным приложением и более чисто разделенной модульной системой.

2. Что уже сделано

2.1. Базовая оболочка приложения

Сделано:
- index.html собран как мобильный application shell;
- есть верхняя панель, основная рабочая область, нижняя навигация;
- подключен manifest.webmanifest;
- подключен service worker sw.js;
- есть поддержка светлой и темной темы;
- приложение поднимается обычным статическим сервером через python3 -m http.server.

2.2. Локальное хранилище и офлайн-режим

Сделано:
- используется IndexedDB;
- в constants определены отдельные store для проектов, классов, учеников, заказов, медиа, операторов, событий операторов, шаблонов чеклистов, настроек, каталога, альбомных сущностей, готовых работ и документов;
- при первом запуске создаются демо-данные;
- service worker кеширует shell и внутренние JS/CSS/иконки.

Это означает, что приложение рассчитано на локальную автономную работу прямо в браузере, без обязательного сервера и без внешней БД.

2.3. Работа с операторами

Сделано:
- есть первичная настройка владельца;
- есть экран выбора сотрудника;
- есть вход по коду;
- есть роли операторов;
- есть включение/отключение операторов;
- есть логирование событий импорта/экспорта и рабочих действий;
- есть статистика по операторам, проектам, группам и активности.

Это уже похоже не просто на личный инструмент, а на систему для команды.

2.4. Проекты, классы, ученики

Сделано:
- создание проектов;
- создание классов/групп;
- сортировка и управление классами;
- создание, редактирование и удаление учеников;
- поиск и переход к карточке ученика;
- хранение статусов оплаты;
- хранение статусов заказа;
- навигация по проектам и группам;
- отдельный экран ученика.

2.5. Каталог услуг и чеклисты

Сделано:
- каталог услуг;
- категории, аудитории и фильтрация;
- карточки услуг;
- детали услуги;
- ракурсы и требования;
- загрузка референсов для услуг и углов;
- массовые операции по услугам;
- популярные услуги;
- шаблон чеклиста;
- применение шаблона к существующим заказам;
- назначение услуги всем ученикам.

Это одна из самых насыщенных по функционалу частей приложения.

2.6. Съемочный поток и медиа

Сделано:
- привязка медиа к ученику;
- импорт медиа с устройства;
- захват фото/видео;
- генерация имен файлов;
- очередь задач по ученику;
- отметка выполнения пунктов чеклиста;
- предпросмотры медиа;
- привязка к оператору, кто снимал/обновлял;
- импорт папок и ZIP для съемки.

2.7. QR-сценарии

Сделано:
- ручной ввод QR;
- переход к ученику по QR;
- сканер QR через камеру;
- fallback через jsQR;
- создание ученика, если QR существует, а локальной записи нет;
- отдельные QR-сценарии для готовых работ и альбомов.

2.8. Блок выпускных альбомов

Сделано:
- отдельные проекты для альбомов;
- группы внутри альбомного проекта;
- ученики альбома;
- общие фото;
- учителя;
- статусы готовности;
- экспорт ZIP по альбомной группе;
- импорт ZIP альбомов;
- PDF-статистика по альбомам;
- QR для альбомных сущностей;
- хранение фото и видео по владельцам.

Это уже не заготовка, а полноценный самостоятельный подсценарий внутри приложения.

2.9. Готовые работы, монтаж и печать

Сделано:

- генерация и хранение итоговых сущностей;
- A4-версии;
- подготовка к печати;
- QR для печатных работ;
- предпросмотр печати внутри приложения;
- экспорт JPG/PDF итогов;
- печать итогов по классу.

2.10. Документы

Сделано:
- загрузка документов;
- привязка документов к проекту, группе или другой сущности;
- предпросмотр;
- скачивание;
- удаление.

2.11. Импорт/экспорт

Сделано:
- полный экспорт данных;
- экспорт проекта;
- экспорт класса;
- экспорт услуг;
- экспорт настроек;
- экспорт рабочей конфигурации;
- импорт этих же типов архивов;
- предпросмотр конфликтов перед импортом;
- стратегии разрешения конфликтов: обновить, оставить старое, создать копию;
- отдельный экспорт/импорт альбомных ZIP;
- экспорт PDF-статистики;
- импорт настроек из ZIP;
- импорт папки услуг.

Это сильная часть проекта. По коду видно, что ей уделено много внимания.

3. Что подтверждено как работающее

Ниже не предположение, а то, что удалось подтвердить по коду и базовой технической проверке.

Подтверждено:
- проект запускается локально через python3 -m http.server на порту 4173;
- index.html отдается корректно с HTTP 200;
- все проверенные JS-файлы проходят синтаксическую проверку через node --check;
- service worker содержит ссылки на реально существующие файлы, явных битых путей в APP_SHELL не найдено;
- package.json минимальный, без конфликтных build-скриптов;
- git status на момент анализа был чистым;
- демо-данные создаются автоматически, если локальная база пуста;
- в коде действительно реализованы маршруты home, classes, scan, albums, settings, student и внутренние дополнительные сценарии;
- логика импорта/экспорта, QR, альбомов, готовых работ, операторов, документов и каталога не выглядит заглушкой, а содержит реальную предметную обработку.

Важно:
полноценного живого браузерного прогона всех экранов я здесь не делал, поэтому ниже я разделяю "подтверждено работает" и "реализовано в коде, но требует ручной проверки по сценариям".

4. Что реализовано, но требует ручной проверки в браузере

С высокой вероятностью это работает, потому что логика в коде есть и она не выглядит пустой, но эти сценарии желательно прогнать руками:

- вход операторов и первичная настройка владельца;
- карточка ученика и полный цикл съемки;
- загрузка фото/видео с конкретных мобильных устройств;
- QR-сканирование через камеру на целевых телефонах;
- импорт и экспорт нескольких реальных архивов;
- блок выпускных альбомов на реальном наборе данных;
- генерация и печать финальных работ;
- предпросмотр документов в разных браузерах;
- работа PWA после установки на устройство;
- восстановление данных после полного импорта архива;
- работа после повторного обновления service worker.

5. Что сейчас не работает, работает нестабильно или имеет явные ограничения

5.1. Нет настоящей модульности приложения

Фактически приложение все еще держится на js/core/legacy-app.js размером более 9000 строк.

Что это означает:
- папки js/modules, js/services, js/ui, js/utils существуют, но большая часть файлов там очень маленькая;
- многие из этих файлов не являются реальными центрами логики приложения;
- основная оркестрация, состояние, рендеры, бизнес-правила, импорт/экспорт и UI-связки живут в одном огромном файле;
- сопровождать и безопасно дорабатывать такой файл тяжело;
- риск регрессий высокий.

Это не "сломано" для пользователя прямо сейчас, но это уже серьезная техническая проблема проекта.

5.2. Нет автоматических тестов

Сейчас в проекте нет:
- unit tests;
- integration tests;
- e2e tests;
- линтера;
- CI-проверок;
- скрипта сборки;
- скрипта валидации качества.

Следствие:
- невозможно быстро доказать, что после изменений ничего не сломалось;
- импорт/экспорт, QR и финальные работы особенно уязвимы к скрытым регрессиям;
- любое развитие проекта будет замедляться.

5.3. Есть зависимость от возможностей конкретного браузера

По коду видны явные ограничения:
- импорт некоторых ZIP зависит от DecompressionStream;
- QR-камера зависит от navigator.mediaDevices;
- быстрый нативный QR зависит от BarcodeDetector;
- часть сценариев печати зависит от поведения окна печати браузера;
- часть сценариев локального импорта завязана на webkitdirectory и файловые возможности браузера;
- IndexedDB и service worker обязательны для полноценной офлайн-работы.

Это означает:
- приложение может вести себя по-разному в Chrome, Safari, Edge и мобильных браузерах;
- особенно чувствительны импорт ZIP и QR-сканирование.

5.4. Уже зашито известное ограничение по Safari и ZIP

В коде есть явная обработка случая:
"Safari не распаковал сжатый ZIP..."

То есть это не теория, а уже признанное ограничение текущей реализации.

Практически это означает:
- часть ZIP-архивов на Safari может не импортироваться напрямую;
- пользователю придется распаковывать архив вручную и использовать обходной путь.

5.5. UX местами держится на prompt/confirm

Во многих сценариях используются:
- prompt;
- confirm;
- простые модальные подтверждения.

Минусы:
- слабый пользовательский опыт;
- неудобно на мобильных;
- сложно расширять;
- сложнее валидировать ввод;
- выше риск случайочного ввода или отмены.

5.6. Нет серверной синхронизации и резервирования вне браузера

Система полностью локальная.

Это плюс для офлайна, но и риск:
- если пользователь потеряет устройство или очистит браузерные данные без экспорта, данные пропадут;
- нет централизованного аккаунта, синхронизации между устройствами или фонового бэкапа;
- перенос данных целиком завязан на ручной импорт/экспорт.

5.7. Архитектура маршрутизации и состояния остается хрупкой

По коду видно:
- много route-состояния хранится вручную;
- много зависимостей между state и DOM;
- много ручных обработчиков событий;
- есть большая связность между экранами.

Следствие:
- любое изменение навигации или состояния может задеть неожиданные сценарии;
- трудно локально изменять один экран без риска побочных эффектов.

6. Что нужно улучшить в первую очередь

6.1. Разбить legacy-app.js на реальные модули

Это главный технический приоритет.

Нужно вынести отдельно:
- маршрутизацию;
- работу с IndexedDB;
- операторов;
- проекты/классы/учеников;
- каталог услуг;
- съемочный поток;
- QR;
- импорт/экспорт;
- альбомы;
- final works;
- печать;
- документы;
- общие UI-компоненты.

Пока это не сделано, любая серьезная доработка будет дорогой и рискованной.

6.2. Добавить минимальный контур качества

Нужно добавить:
- eslint;
- базовые smoke-тесты;
- хотя бы несколько unit-тестов на утилиты и импорт/экспорт;
- несколько e2e-сценариев на Playwright или аналог;
- простую CI-проверку.

Минимальный обязательный набор сценариев для автопроверки:
- первый запуск и создание демо;
- создание проекта, класса и ученика;
- назначение услуги;
- импорт и экспорт;
- переход по QR;
- экспорт альбома;
- создание и экспорт готовой работы.

6.3. Провести реальный кроссбраузерный прогон

Нужно проверить отдельно:
- Chrome Android;
- Safari iPhone/iPad;
- Chrome desktop;
- Edge desktop.

Критические сценарии проверки:
- камера;
- QR;
- ZIP;
- печать;
- офлайн-режим;
- повторный запуск после установки PWA.

6.4. Нормализовать UX форм и диалогов

Нужно постепенно убрать prompt/confirm в пользу собственных форм и модалок.

Это даст:
- более понятный интерфейс;
- валидацию;
- единый стиль;
- меньше случайных ошибок при вводе.

6.5. Усилить безопасность данных

Нужно продумать:
- регулярный экспорт/резервную копию;
- предупреждения о риске потери локальных данных;
- экспорт по расписанию или напоминание;
- явную кнопку бэкапа на главном экране;
- проверку целостности импортируемых архивов.

6.6. Доделать реальную роль папок js/modules и js/services

Сейчас структура уже намекает на хорошую архитектуру, но фактически еще не используется в полном объеме.

Нужно, чтобы:
- модули реально содержали экранную бизнес-логику;
- services реально содержали доменные операции;
- legacy-app перестал быть главным местом всего сразу;
- app.js и core/app.js перестали быть только оболочкой поверх legacy-слоя.

7. Общая оценка текущего состояния

Если оценивать по функциональности:
проект уже сильный и далеко ушел от стадии прототипа.

Если оценивать по архитектуре:
проект перегружен одним большим файлом и уже нуждается в рефакторинге, иначе развитие будет все болезненнее.

Если оценивать по готовности к ежедневной работе:
локально и офлайн он уже выглядит пригодным для использования, особенно если команда понимает ограничения браузеров и регулярно делает экспорт данных.

Если оценивать по готовности к масштабированию:
пока не готов без этапа стабилизации и разбиения монолита.

8. Короткий вывод

Что уже хорошо:
- много реального функционала;
- сильный офлайн-сценарий;
- хорошие импорт/экспорт возможности;
- отдельные рабочие блоки для альбомов, операторов, QR и финальных работ;
- демо-инициализация и локальная автономность.

Что плохо:
- почти вся логика сидит в одном огромном legacy-файле;
- нет тестов;
- есть браузерные ограничения;
- UX местами собран на prompt/confirm;
- надежность изменений пока трудно гарантировать автоматически.

Что делать дальше:
1. Сначала стабилизация и тестовый контур.
2. Затем разбиение legacy-app.js на реальные модули.
3. Затем ручной прогон критических сценариев на нескольких браузерах и устройствах.
4. После этого уже безопасно наращивать новый функционал.

9. На чем основан этот отчет

Отчет основан на:
- чтении структуры репозитория;
- анализе index.html, package.json, sw.js и js-кода;
- проверке запуска локального HTTP-сервера;
- проверке ответа приложения по HTTP 200;
- синтаксической проверке JS-файлов;
- проверке существования файлов, перечисленных в service worker;
- анализе фактической реализации import/export, QR, album, final works, documents и operator flows.

Полный ручной UI-прогон в браузере по всем сценариям в рамках этой проверки не выполнялся, поэтому там, где это важно, я отдельно пометил "требует ручной проверки".
